# Code Review Follow-ups

## 목적
이 문서는 코드베이스 전반을 검토하며 식별한 개선 후보를 정리한 백로그입니다.
기능 변경이 아닌, 코드 품질·유지보수성·안정성 향상을 위한 후속 작업 항목을 관리합니다.

우선순위는 다음 기준으로 분류합니다.
- High: 조속히 개선 권장
- Medium: 다음 리팩토링 사이클에서 검토
- Low: 장기적으로 고려

---

## High Priority

### 1. 테스트 계층 보강
- **대상**: `spec/requests/` 전반
- **문제점**
  - 현재 핵심 동작 검증이 request spec에 집중되어 있습니다.
  - 사용자 흐름 검증에는 적합하지만, 실패 원인 파악과 세부 로직 회귀 탐지에는 한계가 있습니다.

- **개선 이유**
  - 도메인 로직 회귀를 더 빠르게 발견할 수 있습니다.
  - 테스트 실패 원인을 더 명확하게 파악할 수 있습니다.

- **권장 작업**
  - 모델 spec 추가
  - 서비스/쿼리 객체 spec 추가
  - request spec은 사용자 시나리오 중심으로 유지

---

### 2. 도메인 로직 분리 검토
- **대상 후보**
  - 책 검색 및 생성 로직
  - 책장 상태 변경 로직
  - 통계/집계 관련 로직

- **문제점**
  - 일부 도메인 로직이 모델 또는 컨트롤러에 분산될 가능성이 있습니다.

- **개선 이유**
  - 책임 분리가 명확해집니다.
  - 테스트 작성이 쉬워집니다.
  - 재사용성이 향상됩니다.

- **권장 작업**
  - Service Object 또는 Query Object 도입 검토
  - 단, Rails Way를 해치지 않는 범위에서만 적용

---

## Medium Priority

### 1. View Partial 분리
- **대상 후보**
  - `app/views/books/show.html.erb`
  - `app/views/users/show.html.erb`

- **문제점**
  - 화면 복잡도가 증가할수록 템플릿 가독성이 저하될 수 있습니다.

- **개선 이유**
  - 역할별 구조 분리가 가능해집니다.
  - 유지보수성이 향상됩니다.

- **권장 작업**
  - 의미 있는 단위로 partial 분리
  - 과도한 분해는 지양

---

### 2. N+1 점검 강화
- **대상**
  - 목록 조회 액션 전반

- **문제점**
  - 현재 문제는 없어 보이더라도, 향후 기능 확장 시 N+1 위험이 존재합니다.

- **권장 작업**
  - 주요 index/show 액션에 eager loading 재검토
  - Bullet 도입 검토

---

## Low Priority

### 1. Presenter / Decorator 필요성 재평가
- **대상**
  - View Helper가 계속 증가하는 경우

- **검토 이유**
  - 표현 로직이 복잡해질 경우 helper만으로 관리가 어려울 수 있습니다.

- **주의사항**
  - 현재는 도입 필요성이 높지 않음
  - 실제 복잡도가 증가할 때 검토

---

### 2. 성능 측정 기반 최적화 체계 마련
- **권장 작업**
  - 실제 병목이 발생한 후 최적화
  - 추측 기반 최적화 금지

---

## 작업 원칙
- 기능 변경 없이 진행할 것
- 각 작업은 독립 브랜치에서 수행할 것
- 작은 단위로 나누어 진행할 것
- 변경 후 반드시 자동 테스트로 검증할 것

핵심 원칙:

> 더 읽기 쉽게, 더 Rails답게, 더 안전하게.
