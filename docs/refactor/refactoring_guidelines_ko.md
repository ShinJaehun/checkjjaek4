# Refactoring Guidelines

## 목적

이 문서는 프로젝트 전반의 코드 리팩토링 시 일관된 기준을 제공하기 위한
가이드이다. 리팩토링의 목표는 **외부 동작을 변경하지 않고**, 코드의
가독성, 유지보수성, 명확성을 향상시키는 것이다.

> 핵심 원칙: **더 읽기 쉽게, 더 이해하기 쉽게, 더 Rails답게. 단, 동작은
> 100% 동일하게.**

## 주석 작성 원칙

좋은 코드는 주석 없이도 읽혀야 한다. 그러나 다음 경우에는 주석이 가치
있다.

기본 원칙:

-   주석은 코드 설명이 아니라 **의도, 제약, 함정, 주의사항**을 남기는 데
    사용한다.
-   메서드명, 변수명, 구조 개선으로 충분히 풀 수 있는 내용은 주석으로
    해결하지 않는다.
-   리팩토링 과정에서 주석이 필요해졌다면, 먼저 코드 자체를 더 읽기 쉽게
    바꿀 수 없는지 검토한다.
-   주석을 추가했다면 “왜 이 주석이 없으면 미래의 유지보수자가 오해할 수
    있는가”를 설명할 수 있어야 한다.

### 주석이 필요한 경우

-   왜 이렇게 구현했는지 설명할 때
-   Rails의 함정이나 비직관적 동작을 방지할 때
-   미래 유지보수자가 실수하기 쉬운 부분일 때
-   외부 제약사항(성능, 호환성, 보안)이 있을 때

### 좋은 주석 예시

``` ruby
# 저장되지 않은 레코드가 association target에 섞이지 않도록
# form 객체는 association.build 대신 독립적으로 생성한다.
@comment = Comment.new(jjaek: @jjaek)
```

### 피해야 할 주석

``` ruby
# 카운트를 1 증가시킨다
count += 1
```

코드를 그대로 설명하는 주석은 불필요하다.

다음과 같은 주석도 피한다.

-   이미 메서드명이나 조건문이 충분히 설명하는 내용을 반복하는 주석
-   변경 이력이나 임시 메모를 코드에 남기는 주석
-   “나중에 정리”, “일단 이렇게 둠”처럼 의도보다 상태만 기록하는 주석

## 리팩토링 작업 기준

-   리팩토링은 동작 변경이 아니라 **구조 정리와 읽기 쉬움 개선**에 집중한다.
-   하나의 변경은 가능한 한 하나의 목적만 가져야 한다.
-   큰 구조 재설계로 넘어가기 전에, 작은 추출과 중복 제거만으로 충분한지
    먼저 검토한다.
-   helper, private method, local variable 추출은 허용되지만 public interface는
    유지한다.
-   리팩토링 후에는 관련 테스트로 **동작 동일성**을 반드시 확인한다.
