# Bookjjaek Reboot Spec

## 목적

이 문서는 checkjjaek4를 기존의 post 중심 SNS 구조에서
**Jjaek 흐름을 중심으로 하고 책과 서재를 중요한 문맥으로 다루는 독서 커뮤니티 서비스**로
재정의하기 위한 기준 문서다.

이 문서는 아래를 고정한다.

- 제품의 핵심 방향
- 핵심 용어
- 핵심 도메인 모델
- 관계 모델
- 공개 범위 규칙
- Jjaek / ReJjaek / 책활동의 의미
- 향후 구현 시 흔들리면 안 되는 원칙

---

## 제품 방향

bookjjaek은 일반적인 짧은 글 SNS가 아니며,
책과 서재가 중요한 의미를 가지는 독서 커뮤니티다.

bookjjaek의 중심은 다음과 같다.

1. 사용자는 짹(`Jjaek`)을 남긴다.
2. 사용자는 책을 자기 서재에 담는다.
3. 사용자는 필요하면 짹에 책을 연결해 책짹을 남긴다.
4. 사용자는 다른 사람의 책과 짹을 관계에 따라 본다.
5. 사용자는 소식받기와 책친구 관계를 통해 흐름을 이어간다.

즉, 서비스의 중심축은 다음과 같다.

- `Jjaek`
- `User`
- `Follow`
- `BookFriendship`
- `Book`
- `BookshelfEntry`

## 핵심 용어

### 책
서비스 안에서 공통으로 참조되는 도서 정보.
개인 감정, 상태, 공개 범위는 책 자체에 저장하지 않는다.

### 서재
사용자가 자기 책 목록과 독서 상태를 쌓는 **전체 독서 공간**.
초기 MVP에서는 사용자마다 기본 서재 1개를 전제로 한다.

### 책장
서재 안에 들어가는 **하위 컬렉션 단위**.
후속 단계에서는 한 서재 안에 여러 책장을 둘 수 있으며,
각 책장은 공개/비공개 설정을 가질 수 있다.

초기 MVP에서는 별도 책장 모델을 도입하지 않고,
사용자마다 기본 서재 1개와 그 안의 기본 책장 1개처럼 단순화해 해석한다.

### 서재 항목
특정 사용자가 특정 책을 자기 서재 안 어떤 책장/목록 문맥으로 가지고 있는지를 나타내는 기록.
초기 MVP에서는 기본 서재/기본 책장 기준으로 해석한다.

### 대표 상태
한 서재 항목이 가지는 기본 상태.
초기 버전에서는 아래 3개만 허용한다.

- 읽고 싶어요
- 읽는 중
- 읽었어요

대표 상태는 한 서재 항목당 1개만 가진다.

### 스티커
대표 상태와 별개로, 책에 대한 감정이나 의도를 표현하는 보조 태그.
한 서재 항목에 여러 개를 붙일 수 있다.

예:
- 좋았어요
- 너무 좋았어요
- 마음에 남아요
- 다시 읽고 싶어요
- 추천받았어요
- 추천하고 싶어요

### Jjaek
사용자가 남기는 기본 콘텐츠 단위.
Jjaek은 책 없이도 존재할 수 있다.
필요하면 특정 책에 연결될 수 있다.

예:
- 일반 메시지
- 사진이 포함된 짧은 글
- 링크/인용구 공유
- 저자에 대한 토론
- 한줄평
- 짧은 감상
- 긴 감상
- 쓰레드형 글

책이 연결된 `Jjaek`는 UI 문맥에서 **책짹**이라 부를 수 있다.

### ReJjaek
다른 사용자의 Jjaek을 인용하고,
그 위에 내 의견을 덧붙여 새 Jjaek을 만드는 기능.
성격상 책 상태 기록보다 **인용/응답**에 더 가깝다.

UI 문맥에서 **다시짹**이라 부를 수 있다.

### 책활동
Jjaek처럼 본문이 있는 글이 아니라,
서재 상태 변경이나 스티커 추가처럼 발생하는 활동 기록.

예:
- 어떤 책을 읽고 싶어요로 담음
- 어떤 책을 읽는 중으로 바꿈
- 어떤 책에 너무 좋았어요 스티커를 붙임

초기 MVP에서는 책활동을 사용자-facing 개념으로 유지하되,
별도의 피드 이벤트 모델(`BookActivity`) 도입은 후속 단계로 미룬다.

### BookActivity
`BookActivity`는 책과 관련된 사용자 행동을
피드에서 소비하기 좋은 형태로 기록하는 **피드용 이벤트 모델**이다.

예를 들면 아래 같은 사건을 담을 수 있다.

- 책을 서재에 처음 담음
- 대표 상태를 읽고 싶어요 → 읽는 중으로 변경
- 특정 스티커를 추가하거나 제거

중요한 점:
- `BookActivity`는 책에 대한 본문 글이 아니다.
- `BookActivity`는 `BookshelfEntry`나 `BookshelfEntrySticker` 자체를 대체하지 않는다.
- `BookActivity`의 목적은 **도메인 상태 저장**이 아니라
  **피드 노출용 이벤트 표현**이다.

### 소식받기
한 사용자가 다른 사용자의 공개 Jjaek을
자기 피드에서 받고 싶을 때 맺는 가벼운 단방향 관계.
기술적으로는 follow 개념과 대응한다.

### 책친구
신청/수락을 통해 성립하는 신뢰 관계.
책친구 공개 Jjaek을 볼 수 있는 기준이 된다.

---

## 관계 모델

### 소식받기
의미:
- 이 사람의 공개 Jjaek을 내 피드에서 보고 싶다.

특징:
- 단방향 관계다.
- 책친구와 다르다.
- 공개 콘텐츠를 피드에 띄우는 기준이다.
- 프로필 화면에서는 **공개 짹까지 볼 수 있게 하는 가벼운 관계**로 해석할 수 있다.

### 책친구
의미:
- 이 사람과는 좀 더 안심하고 독서 기록과 Jjaek을 주고받을 수 있다.

특징:
- 신청/수락이 필요하다.
- 소식받기와 다르다.
- 책친구 공개 범위를 볼 수 있는 기준이다.
- 프로필 화면에서는 **더 넓은 짹 열람 범위와 상호작용을 여는 신뢰 관계**로 해석한다.

## 공개 범위

Jjaek의 공개 범위는 아래 3개만 둔다.

- 전체 공개
- 책친구 공개
- 나만 보기

### 전체 공개
누구나 볼 수 있다.

### 책친구 공개
작성자의 책친구만 볼 수 있다.

### 나만 보기
작성자만 볼 수 있다.

## 대표 상태와 스티커의 구분

대표 상태와 스티커는 같은 것이 아니다.

### 대표 상태
책의 현재 위치를 나타낸다.

- 읽고 싶어요
- 읽는 중
- 읽었어요

### 스티커
그 책에 대한 감정, 인상, 의도를 표현한다.

- 좋았어요
- 너무 좋았어요
- 마음에 남아요
- 다시 읽고 싶어요
- 추천받았어요
- 추천하고 싶어요

원칙:
- 대표 상태는 1개만 가진다.
- 스티커는 여러 개 가질 수 있다.

## Jjaek과 책활동의 구분

### Jjaek
사용자가 직접 쓰는 글이다.
본문이 있다.
책에 선택적으로 연결될 수 있다.
댓글/좋아요/ReJjaek의 대상이 된다.

### 책활동
서재 상태나 스티커 변경으로 생기는 활동이다.
본문이 필수는 아니다.
피드를 풍성하게 만드는 이벤트다.

원칙:
- 도메인 상 Jjaek과 책활동은 분리한다.
- 사용자에게는 필요 이상으로 복잡하게 드러내지 않는다.
- 피드에서는 둘을 하나의 흐름처럼 함께 보여줄 수 있다.
- 초기 MVP에서는 우선 Jjaek 중심 피드부터 완성하고,
  책활동용 이벤트 모델은 후속 단계에서 도입한다.

### BookshelfEntry / BookshelfEntrySticker / BookActivity 책임 경계

#### BookshelfEntry
- 사용자가 특정 책을 어떤 상태로 가지고 있는지 저장하는 **현재 상태 모델**이다.
- 대표 상태의 source of truth다.
- 피드 이벤트 자체를 표현하지 않는다.

#### BookshelfEntrySticker
- 특정 서재 항목에 어떤 스티커가 붙어 있는지 저장하는 **연결 모델**이다.
- 스티커 부착 상태의 source of truth다.
- 피드 이벤트 자체를 표현하지 않는다.

#### Jjaek
- 사용자가 직접 쓴 **본문 글 모델**이다.
- 책 없이도 존재할 수 있고, 필요하면 특정 책에 연결될 수 있다.
- 감상, 생각, 인용, ReJjaek을 담는다.
- 댓글/좋아요/ReJjaek의 대상이 된다.

#### BookActivity
- `BookshelfEntry` / `BookshelfEntrySticker` 변화에서 파생될 수 있는
  **피드용 이벤트 모델**이다.
- 어떤 변화가 있었는지를 시간순으로 보여주기 위한 표현 계층이다.
- 상태의 source of truth가 아니다.
- 책에 대한 자유 본문 글도 아니다.

즉:
- 현재 상태는 `BookshelfEntry`
- 현재 스티커 부착 상태는 `BookshelfEntrySticker`
- 본문 글은 `Jjaek`
- 피드용 활동 이벤트는 나중의 `BookActivity`

## ReJjaek 규칙

ReJjaek은 새 Jjaek이지만,
원문 권한을 절대 넘어서면 안 된다.

### 기본 규칙

1. ReJjaek은 원문보다 넓게 공개할 수 없다.
2. 원문이 나만 보기이면 ReJjaek할 수 없다.
3. ReJjaek 가능 여부의 핵심 기준은 **BookshelfEntry 존재 여부가 아니라 원문 접근 권한**이다.
4. ReJjaek은 원문 내용을 복사 저장하지 않고 참조한다.
5. ReJjaek 조회 시에도 원문 접근 권한을 다시 검사한다.
6. 원문 접근 권한이 사라지면 원문 내용과 quoted block을 더 이상 노출하지 않는다.
7. 초기 MVP에서는 구현 단순성과 안전성을 우선하여,
   원문 접근 권한이 사라진 ReJjaek은 **전체 비노출**을 기본 정책으로 둔다.
8. 후속 확장에서는 ReJjaek 본문은 남기고 quoted block만
   `원문을 볼 수 없음` 상태로 대체하는 UI를 검토할 수 있다.
9. 초기 MVP에서는 구현 단순성과 권한 일관성을 위해
   ReJjaek이 다른 ReJjaek을 다시 인용하는 **중첩 ReJjaek**은 허용하지 않는다.
10. ReJjaek의 1차 작성 진입 UX는 아래 `ReJjaek 1차 UX` 기준을 따른다.
    이를 별도 모델로 곧바로 분리하지는 않으며,
    우선 `Jjaek`의 `quoted_jjaek` 문맥으로 유지한다.

### 예시

원문이 전체 공개:
- ReJjaek 공개 범위: 전체 공개 / 책친구 공개 / 나만 보기 가능

원문이 책친구 공개:
- ReJjaek 공개 범위: 책친구 공개 / 나만 보기만 가능
- 전체 공개 불가

원문이 나만 보기:
- ReJjaek 불가

## 댓글과 좋아요 원칙

초기 버전에서는 댓글과 좋아요의 대상은 `Jjaek`으로 한정한다.

원칙:
- 댓글/좋아요는 **해당 Jjaek을 볼 수 있는 사람만** 가능하다.
- 즉, 상호작용 가능 범위는 공개 범위보다 넓어질 수 없다.
- 초기 MVP에서는 작성자가 개별 Jjaek마다 댓글 허용 여부를 따로 조절하는 기능은 두지 않는다.

### 댓글
- 전체 공개 Jjaek: 로그인 사용자는 허용
- 책친구 공개 Jjaek: 책친구만 허용
- 나만 보기 Jjaek: 작성자만 허용

### 좋아요
- 전체 공개 Jjaek: 로그인 사용자는 허용
- 책친구 공개 Jjaek: 책친구만 허용
- 나만 보기 Jjaek: 작성자만 허용

---

## 피드 원칙

피드는 서비스의 중심 모델이 아니라
도메인 모델이 만들어내는 결과물이다.

초기 MVP의 피드는 아래를 우선 대상으로 한다.

- 소식받는 사람의 전체 공개 Jjaek
- 책친구의 책친구 공개 Jjaek

후속 확장에서는 아래를 추가할 수 있다.

- 소식받는 사람의 전체 공개 책활동
- 책친구의 책친구 공개 책활동

원칙:
- 피드는 `Post` 단일 테이블 중심으로 설계하지 않는다.
- 장기적으로는 Jjaek과 책활동을 함께 다룰 수 있게 설계한다.
- 하지만 초기 구현은 Jjaek 중심 피드부터 시작한다.

## 1차 MVP 화면 경계

초기 MVP에서는 각 화면의 책임을 작게 유지한다.
하나의 화면이 너무 많은 쓰기 기능과 보조 위젯을 동시에 가지지 않도록 한다.

### 홈
- 홈은 **Jjaek 흐름 중심 화면**으로 둔다.
- 초기 홈에는 아래만 우선 포함한다.
  - 내가 남긴 짹
  - 소식받는 사람의 전체 공개 짹
  - 책친구의 책친구 공개 짹
  - 사람 문맥에서 남긴 visible `profile-context Jjaek`
  - 일반 `짹` 입력창
  - 책 검색 진입
  - 내 서재 보기 진입
- 아래 요소는 초기 홈의 필수 범위에 포함하지 않는다.
  - 책친구 요청 위젯
  - BookActivity 요약 위젯
  - 그룹 관련 위젯

### 프로필
- 프로필(`users/:id`)은 **관계 맺기 + 관계에 따라 그 사람의 책과 짹을 보는 화면**으로 둔다.
- 프로필에는 해당 사람을 문맥으로 삼는 `profile-context Jjaek` 입력 진입을 둘 수 있다.
- 이 입력은 DM이 아니라, `target_user_id`를 가지는 공개/관계기반 `Jjaek` 작성이다.
- 이 문맥의 `Jjaek`는 `A가 B에게 남긴 짹`처럼 표시할 수 있다.
- 관계에 따라 아래 노출 범위를 다르게 둔다.
  - 아무 관계 없음:
    - 공개 책장에 담긴 책 목록만 본다
    - 공개 짹은 보지 않는다
    - 책친구 공개 짹은 보지 않는다
    - profile-context 입력창은 보지 않는다
  - 소식받기(`Follow`)만 함:
    - 공개 책장에 담긴 책 목록을 본다
    - 공개 짹을 본다
    - 책친구 공개 짹은 보지 않는다
    - profile-context 입력창은 보지 않는다
  - accepted 책친구:
    - 공개 책장에 담긴 책 목록을 본다
    - 공개 짹과 책친구 공개 짹을 본다
    - profile-context 입력창을 본다
  - 본인:
    - 자신의 전체 서재와 전체 짹을 본다
    - profile-context 입력창을 포함한 모든 작성 진입을 사용할 수 있다

### 서재
- 서재는 **내 독서 공간 전체**를 뜻한다.
- 초기 MVP에서는 기본 서재 1개를 전제로 한다.
- 서재 화면은 **내 책 목록과 상태/스티커 관리**에 집중한다.
- 현재 MVP에서는 별도 책장 모델을 두지 않고,
  기본 서재 안의 기본 책장 1개처럼 단순화해 해석한다.
- 따라서 이번 단계의 최소 구현에서는 타인 프로필의 현재 `BookshelfEntry` 목록 전체를
  "기본 공개 책장"으로 임시 해석한다.
- 이 규칙은 여러 책장/책장 공개범위 모델 도입 전까지의 MVP 임시 규칙이다.
- 서재 화면은 책짹을 책/서재 문맥에서 다시 모아보는 진입점이 될 수 있다.

### 책 상세
- 책 상세(`books/:id`)는 **책 문맥 화면**으로 둔다.
- 책 상세에는 아래 요소를 함께 둔다.
  - 책 메타데이터
  - 그 책에 연결된 visible 책짹 목록
  - 책짹 입력창
  - 대표 상태 선택
  - 스티커 선택
- 즉, 초기 MVP에서는 책 상세 안에서
  **서재 수정과 책짹 작성이 함께 일어날 수 있다.**
- 다만 내부 도메인 저장 책임은 계속 분리한다.
  - 상태/스티커 저장은 `BookshelfEntry`
  - 책짹 저장은 `Jjaek`
- 같은 화면에 공존하더라도 저장 책임은 합치지 않는다.
  - 상태/스티커 저장은 별도 form
  - 책짹 작성은 별도 form
  - 저장 엔드포인트도 분리한다
- 다만 사용자가 검색 결과의 `책 상세 보기`를 통해 먼저 들어올 수 있으므로,
  `books/:id` 접근 자체는 `BookshelfEntry`가 없어도 허용할 수 있다.
- 이 경우 책 상세는 **읽기 전용 상태**로 동작한다.
  - 책 메타데이터는 볼 수 있다.
  - 다른 사람의 visible 책짹 목록은 볼 수 있다.
  - 하지만 상태/스티커 저장과 책짹 작성은 열리지 않는다.
- 즉, 상태/스티커 변경과 Jjaek 작성은
  **현재 사용자의 BookshelfEntry가 있을 때만** 가능하다.
- 같은 화면의 입력 영역 버튼 문구는 `책짹`으로 둔다.
- 책 상세에 관계 위젯, 홈 요약 위젯, BookActivity 요약까지 함께 싣는 복합 대시보드는 두지 않는다.

### Jjaek 작성
- 일반 Jjaek 작성의 기본 진입은 홈에 둔다.
- 책이 연결된 Jjaek 작성의 기본 진입은 책 상세 안에 둔다.
- UI 문맥상 아래처럼 부를 수 있다.
  - 일반 작성: `짹`
  - 책이 연결된 작성: `책짹`
  - 인용/응답 작성: `다시짹`
- 세 경우 모두 기본 모델은 `Jjaek`로 유지한다.

### Jjaek 표시 규칙 1차

- 일반 `Jjaek`는 기본적으로 `짹`으로 표시할 수 있다.
- `target_user_id`가 있는 `Jjaek`는 `A가 B에게 남긴 짹`처럼 표시할 수 있다.
- `book_id`가 있는 `Jjaek`는 `A가 X에 대해 남긴 책짹`처럼 표시할 수 있다.
- `quoted_jjaek_id`가 있는 `Jjaek`는 `A가 B의 짹을 다시짹`처럼 표시할 수 있다.
- `target_user_id`와 `quoted_jjaek_id`가 함께 있는 경우에도, 1차 MVP에서는 `다시짹` 문맥을 우선해 표시할 수 있다.
- `book_id`와 `quoted_jjaek_id`가 함께 있는 경우에도, 1차 MVP에서는 `책짹`보다 `다시짹` 문맥을 우선해 표시할 수 있다.
- `book_id`와 `target_user_id`가 함께 있는 경우에는 1차 MVP에서 `책짹` 문맥을 우선하고,
  사람 문맥은 후속 단계에서 더 세분화할 수 있다.
- 이 표시 규칙은 UI용 해석이며, 기본 모델은 계속 `Jjaek`로 유지한다.

### ReJjaek 1차 UX

- ReJjaek은 별도 모델이 아니라 `Jjaek`의 `quoted_jjaek` 문맥으로 해석한다.
- 1차 MVP의 다시짹 작성 진입은 `jjaeks/:id` 상세 화면에 둔다.
- 다시짹은 책짹의 하위 기능이 아니라, 특정 짹에 대한 인용/응답 작성이다.
- 원문이 private이거나 원문 자체가 ReJjaek이면 다시짹 작성 진입을 노출하지 않는다.
- 다시짹은 원문보다 넓은 공개 범위로 작성할 수 없다.
- 원문 접근 권한이 사라지면 다시짹도 조회되지 않아야 한다.
- books/:id, home feed, users/:id 카드 액션은 후속 단계에서 검토한다.

### 검색 결과
- 검색 결과는 우선 **책 상세 진입**을 기본 흐름으로 둔다.
- 검색 결과에서 바로 여러 쓰기 동작을 수행하지 않고, 책 상세로 진입한 뒤
  상태/스티커/책짹 상호작용을 수행한다.
- 다만 검색 결과 화면에서는 **`서재에 담기`** 용어를 사용한다.
- 여기서 `서재에 담기`는 외부 검색 결과의 책 정보를
  우리 서비스의 DB/서재로 가져오는 **첫 등록(import) 동작**을 뜻한다.
- 즉:
  - 검색 결과 버튼: `서재에 담기`
  - 이 동작으로 책이 우리 DB와 내 서재에 처음 등록된다
  - 그 다음부터는 `books/:id`에서 상태/스티커와 Jjaek을 다룬다

### 초기 MVP UI 용어
- 검색 결과 버튼: `서재에 담기`
- 홈의 일반 작성 버튼: `짹`
- 책 상세의 책 연결 작성 버튼: `책짹`
- 인용/응답 작성 버튼: `다시짹`
- 책 상세의 상태/스티커 저장 버튼: `저장` 또는 `상태 저장`

### 홈 피드와 책활동
- 초기 MVP의 홈 피드는 계속 **Jjaek 중심 피드**로 유지한다.
- 상태/스티커 변경은 책 상세 안에서 수행할 수 있지만,
  이를 홈 피드 이벤트로 반영하는 `BookActivity`는 아직 도입하지 않는다.
- 즉, 초기 MVP에서는 상태/스티커 저장이 `BookshelfEntry`를 갱신하더라도
  홈 피드에는 별도 활동 이벤트를 추가하지 않는다.

왜 초기 MVP에서 `BookActivity`를 구현하지 않는가:
- 우선순위가 **책 상태 관리 자체**보다 **피드 이벤트 모델링**으로 옮겨가 버릴 수 있기 때문이다.
- `Jjaek` 피드만으로도 초기 사용자 가치 검증이 가능하기 때문이다.
- 상태 변경 저장과 피드 이벤트 저장을 동시에 도입하면
  생성 규칙, 중복 처리, 노출 정책이 한 번에 복잡해지기 때문이다.
- 아직은 “상태/스티커를 어떻게 저장하고 수정하는가”를 먼저 고정하는 단계이기 때문이다.

후속 단계에서 `BookActivity`를 도입하면 맡게 될 역할:
- `BookshelfEntry` 생성/변경/스티커 변경을 피드에서 읽기 좋은 이벤트로 표현
- 홈 피드에서 `Jjaek`과 함께 한 흐름으로 섞여 보이게 하는 기반 제공
- “책을 담음”, “읽는 중으로 변경”, “스티커를 붙임” 같은 활동을
  본문 글과 분리된 형태로 노출

## 향후 확장: 한 서재 안 여러 책장 지원

초기 MVP에서는 사용자마다 기본 서재 1개를 전제로 한다.

즉:
- 사용자는 자기 서재에 책을 담는다.
- 책 상태와 스티커는 기본 서재 기준으로 관리한다.
- 별도의 커스텀 책장 모델은 도입하지 않는다.

향후 확장에서는 한 서재 안에 여러 개의 사용자 정의 책장을 지원할 수 있다.

이 확장 단계에서는:
- 한 사용자가 여러 책장을 가질 수 있다.
- 각 책장은 공개 여부를 가질 수 있다.
  - 공개 책장
  - 비공개 책장
- 프로필 화면에서 다른 사용자가 보는 책 목록은 우선 **공개 책장** 기준으로 해석한다.
- 본인은 자신의 공개/비공개 책장을 모두 볼 수 있다.

예:
- 공개 책장
- 비공개 책장

이 확장 단계에서는 아래 모델 도입을 검토할 수 있다.

- Bookshelf
- BookshelfMembership
- BookshelfEntry

여기서 한국어 용어 해석은 다음과 같이 둔다.
- 서재 = 사용자 전체 독서 공간
- Bookshelf = 서재 안의 책장

## 향후 확장: 그룹 기능

그룹 기능은 가치 있는 확장 방향이지만,
초기 MVP 범위에는 포함하지 않는다.

초기 MVP에서는 개인 서재와 개인 간 관계를 먼저 완성한다.

향후 확장에서는 아래와 같은 그룹 기능을 지원할 수 있다.

- 학급 그룹
- 독서모임 그룹
- 프로젝트 그룹
- 그룹 안 독서 활동
- 그룹 안 Jjaek
- 그룹 기반 사용자에서 일반 사용자로의 전환 흐름

이 확장 단계에서는 아래 모델 도입을 검토할 수 있다.

- Group
- GroupMembership

## 향후 확장: 자동 생성 학생 계정과 일반 사용자 전환

교사가 자동 생성한 학생 계정은
처음에는 그룹 기반 관리형 계정일 수 있다.

하지만 사용자가 원하면
기존 활동 기록을 유지한 채 일반 사용자 계정으로 전환할 수 있어야 한다.

원칙:
- 새 계정을 따로 만드는 것보다 기존 계정을 승격하는 방향을 우선한다.
- 기존 서재와 Jjaek 기록은 유지한다.
- 그룹 기반 사용에서 개인 서재 기반 사용으로 이어질 수 있어야 한다.

이 기능 역시 초기 MVP 범위에는 포함하지 않고,
그룹 기능과 함께 후속 단계에서 검토한다.

## 설계 원칙 요약

1. 책은 공용 정보다.
2. 서재는 사용자 전체 독서 공간이다.
3. 책장은 서재 안의 하위 컬렉션이다.
4. 대표 상태는 1개, 스티커는 여러 개다.
5. Jjaek은 사용자의 기본 콘텐츠 단위이며, 책은 선택적으로 연결된다.
6. 소식받기와 책친구는 다른 관계다.
7. 공개 범위는 전체 공개 / 책친구 공개 / 나만 보기로 제한한다.
8. ReJjaek은 원문 권한을 넘을 수 없다.
9. 피드는 도메인의 결과물이지 중심 모델이 아니다.
10. 초기 MVP에서는 개인 서재와 개인 간 관계를 먼저 완성한다.
11. 한 서재 안 여러 책장과 책장 공개/비공개는 후속 확장으로 남긴다.
12. 그룹, 학생 계정 전환 흐름은 후속 확장으로 남긴다.
